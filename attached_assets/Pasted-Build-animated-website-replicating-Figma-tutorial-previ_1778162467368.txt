Product Requirements Document: Animated Character Showcase
1. Executive Summary
Build a full-viewport, desktop-first animated landing page that cycles through four distinct 3D characters. The experience must replicate the exact motion language, timing, and visual hierarchy demonstrated in the Figma tutorial preview: large clipped typography as a background layer, a central hero character render, and coordinated slide transitions for all UI elements.

3. Animation & Interaction Specification
3.1 Core Transition Mechanics
The site functions as a 4-step carousel. Transitions are triggered by:
Auto-play: Advance every 4 seconds.
Manual: Click/tap anywhere on the viewport (or on a hidden click target) to advance.
Keyboard: Arrow keys (Right/Down = next, Left/Up = previous).
3.2 The "Clip Content" Technique
To perfectly match the Figma Smart Animate behavior, implement overflow-hidden masks for sliding content:
A. Background Name Scroll
Container: A full-viewport div with overflow: hidden.
Content Track: A vertical flex column containing all four names stacked. Each name is the full viewport height.
Motion: When changing slides, translate the track Y by -100% × currentIndex.
Duration: 1200ms.
Easing: cubic-bezier(0.76, 0, 0.23, 1) (Custom Bezier from Figma prototype).
B. Character Image Swap
Container: Centered div, overflow: visible (or hidden depending on desired crop), fixed size.
Motion: Use a cross-fade + slight horizontal slide.
Outgoing: opacity: 1 → 0, translateX: 0 → -5%, scale: 1 → 0.95.
Incoming: opacity: 0 → 1, translateX: 5% → 0, scale: 1.05 → 1.
Duration: 1200ms (overlap with background text).
Easing: Same custom bezier.
C. Info Panel Text Scroll
Container: overflow: hidden, fixed height (~120px).
Content Track: Vertical stack of 4 text blocks (Name + Description).
Motion: Translate Y by -100% × currentIndex, synchronized with background text.
Duration: 1200ms.
Easing: Same custom bezier.
D. Thumbnail Image Scroll
Container: overflow: hidden, border-radius: 12px, fixed size (120×176px).
Content Track: Vertical stack of 4 thumbnail images.
Motion: Translate Y by -100% × currentIndex, synchronized.
Duration: 1200ms.
E. Background Color Morph
Technique: Two full-screen background layers. The incoming layer sits above the outgoing layer.
Motion: Incoming layer fades in (opacity: 0 → 1) over 800ms with a slight scale (scale: 1.05 → 1) to avoid dead frames.
Alternative: Animate a CSS background-color transition on a single layer (800ms ease).

4.2 Asset Requirements
Hero Images: 4 high-resolution PNGs (or WebP with alpha) of 3D characters. Each character must be isolated on a transparent background, centered, full-body or bust-up, facing slightly toward camera. Style: Felted/wool texture, Pixar-esque 3D renders, warm lighting.
Thumbnails: 4 cropped JPG/PNG versions (portrait ratio, 3:4), showing the character's torso/outfit detail.
Texture: A subtle static noise PNG tile for the grain overlay.
5. Technical Architecture
5.1 Recommended Stack
Framework: React 18+ (Next.js App Router optional).
Animation: Framer Motion is strongly recommended because:
AnimatePresence handles the enter/exit choreography cleanly.
layoutId can be used for shared element transitions if needed.
Custom spring/bezier support (transition: { duration: 1.2, ease: [0.76, 0, 0.23, 1] }).
Alternative: GSAP with Flip plugin + Timeline for maximum control.
Styling: Tailwind CSS or CSS Modules. No UI component libraries (the design is bespoke).

5.3 Critical Implementation Notes
Clip Content / Overflow Hidden: The "magic" of the Figma tutorial relies entirely on masked frames. In CSS, this means every sliding element (names, info, thumbnails) must live inside a parent with overflow: hidden and explicit dimensions.
Hardware Acceleration: All sliding motions must use transform: translate3d(...) to force GPU compositing and avoid jank on the large typography.
Will-Change: Apply will-change: transform, opacity to the sliding tracks during transitions.
Preloading: Hero PNGs must be preloaded to prevent pop-in during the first transition.
Auto-Play Logic:
useEffect interval set to 4000ms.
Pause on hover (optional but recommended).
Reset timer on manual navigation.
6. Responsive Behavior
Primary Target: Desktop (1920×1080 and 1440×900). The design is explicitly widescreen.
Tablet (≤1024px): Scale down typography to 35vw, reduce character height to 65vh, reposition bottom panels to bottom: 40px, left/right: 40px.
Mobile (≤768px):
Background text becomes 40vw.
Character image becomes 60vh and shifts down slightly.
Right tagline moves below the character or is hidden.
Info panel and thumbnail stack vertically at bottom center.
Note: The original design is desktop-centric; mobile is a graceful degradation.
7. Acceptance Criteria (Definition of Done)
[ ] Visual Fidelity: Background colors, font weights, spacing, and border radii match the video frames within 5% tolerance.
[ ] Typography Mask: The large background text has a gradient mask fading to transparent at the bottom, allowing the character to visually sit "in front" while remaining readable at the top.
[ ] Transition Timing: All sliding elements (names, info, thumbnails) move at exactly 1200ms using the custom bezier cubic-bezier(0.76, 0, 0.23, 1).
[ ] Staggered Character Entrance: The incoming hero image should not appear instantly; it must fade/slide in with a slight delay (~100-200ms) after the transition begins.
[ ] Looping: After the 4th slide (Sunny), the next click/interval returns to the 1st slide (Norish) seamlessly.
[ ] No Layout Shift: During transitions, the viewport must not scroll or resize. All motion is contained within overflow: hidden bounds.
[ ] Performance: Maintain 60fps during transitions. Use Chrome DevTools Performance panel to verify.
[ ] Asset Safety: If images fail to load, show a solid-color fallback matching the slide's bgColor.
